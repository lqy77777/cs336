# 如何实现 3.5 Pre-Norm Transformer Block & Transformer LM(思路提示)

> 这是实现思路的提示,不含具体代码。具体怎么写需要自己完成。

## 这一节在整体架构里的位置

3.5 其实是两个独立的 Problem,把前面所有零件组装起来:

1. **`transformer_block`**:一个 Transformer block = 一个多头自注意力子层 + 一个 SwiGLU FFN 子层,用 pre-norm 的方式接起来,外加残差连接。这一步用到你已经写好的 `RMSNorm`、`multihead_self_attention`(带 RoPE 版本)、`Feedforward`。
2. **`transformer_lm`**:把 `num_layers` 个 Transformer block **堆叠**起来,前面接 token embedding,后面接最终的 RMSNorm 和一个把 `d_model` 映射到 `vocab_size` 的输出层(`lm_head`),得到每个位置的 logits。

## Part 1: Pre-Norm Transformer Block

### 公式(讲义给出的第一子层)

$$
y = x + \text{MultiHeadSelfAttention}(\text{RMSNorm}(x))
$$

注意这里的顺序:**先 norm,再算子层,最后把子层的输出加回原始、没有被 norm 过的 `x`**——这就是"pre-norm"名字的由来(norm 在子层之前,而不是像原始 Transformer 论文那样在子层之后)。讲义提到这样做的直觉是:保留一条从输入 embedding 到最终输出、完全不经过任何 normalization 的"干净残差流",有利于梯度传播。

第二子层(FFN)遵循**完全相同的模式**,只是把 `MultiHeadSelfAttention` 换成 `Feedforward`:

$$
z = y + \text{Feedforward}(\text{RMSNorm}(y))
$$

(讲义正文提到"两个子层"都遵循这个 pre-norm + 残差的结构,配图是 Figure 2,建议对照图看一下这个数据流向,加深理解。)

### 需要想清楚的点

- 一个 block 内部需要几个 `RMSNorm` 实例?两个子层各自的 norm 是共享同一份权重,还是各自独立学习?(提示:norm 里的 `weight` 是可学习参数,两个子层归一化的输入分布完全不同,该不该用同一组参数?)
- 你的 `multihead_self_attention` 现在应该已经支持 RoPE 了(上一轮我们讨论过 `theta`/`max_seq_len`/`token_positions` 怎么接进去)。这里 `TransformerBlock` 的构造函数要不要接收 `max_seq_len`、`theta` 这两个参数,再转手传给内部的注意力子层?
- `token_positions` 在这一层要不要作为 `forward` 的参数暴露出去,还是内部自己算一个默认值(回忆之前讨论的"没有 padding 场景下,默认就是 `0,1,2,...,seq_len-1`")?

### 一个重要的隐藏线索:属性命名

去翻一下 `tests/adapters.py` 里 `run_transformer_block` 的 docstring,它列出了参考实现的 `weights` 字典的 key,比如:

```
attn.q_proj.weight
attn.k_proj.weight
attn.v_proj.weight
attn.output_proj.weight
ln1.weight
ffn.w1.weight
ffn.w2.weight
ffn.w3.weight
ln2.weight
```

回忆你之前遇到过的 `load_state_dict` 的点路径命名规则(嵌套子模块的参数名 = `子模块属性名.参数名`,层层拼接)。这些 key 其实在暗示参考实现内部子模块的**属性名**是什么(比如 `self.attn`、`self.ln1`、`self.ffn`、`self.ln2`,而 `attn` 内部又有 `self.q_proj`、`self.k_proj`、`self.v_proj`、`self.output_proj`)。

你现在 `multihead_self_attention` 里用的是 `self.W_Q`/`self.W_K`/`self.W_V`/`self.W_O`,`Feedforward` 里用的是 `self.W_1`/`self.W_2`/`self.W_3`——跟这里的命名对不上。这不代表你之前写错了,只是意味着写 `run_transformer_block` 这个 adapter 的时候,你要么:

1. 把自己模块内部的属性名改成和这份 key 一致,这样就能直接 `block.load_state_dict(weights)`;或者
2. 保持自己原来的命名,在 adapter 函数内部手写一次 key 的重映射(构造一个新 dict,把 `attn.q_proj.weight` 转成你自己模块认识的 key),再调用 `load_state_dict`。

两种做法都可行,选哪个是你自己的设计决定——但这是一个值得提前想清楚的点,不然写 adapter 的时候会卡在"权重加载不进去"上。

## Part 2: 完整的 Transformer LM

### 结构(对照 Figure 1)

```
token_ids  →  Embedding  →  TransformerBlock × num_layers(依次堆叠)→  RMSNorm(ln_final)  →  Linear(lm_head)  →  logits
```

- 输入是 token id(整数),不是向量,所以第一步是你写好的 `Embedding` 模块。
- 中间堆叠 `num_layers` 个 `TransformerBlock`,每一个的输出是下一个的输入。
- 最后再过**一次额外的** `RMSNorm`(讲义原文提到"pre-norm Transformer... with an additional layer normalization after the final Transformer block")——注意这是整个模型层面的最后一次 norm,不属于任何一个 block 内部。
- `lm_head`:把 `d_model` 维映射到 `vocab_size` 维,得到每个位置在整个词表上的**未归一化** logits(不要在这里做 softmax——讲义原文说 logits 是 "unnormalized distribution over the vocabulary",归一化留给后面 4.1 节的 cross-entropy 去做)。

### 一个新概念:怎么存 `num_layers` 个子模块?

你需要在 `__init__` 里创建 `num_layers` 个 `TransformerBlock` 实例。如果直接用 Python 原生的 `list` 存(比如 `self.layers = [TransformerBlock(...) for _ in range(num_layers)]`),会有什么问题?

回忆 `nn.Module` 是怎么"发现"自己有哪些可学习参数的——它靠的是 `__setattr__` 里对 `nn.Parameter`/`nn.Module` 类型的特殊处理(你在 `Linear`/`RMSNorm` 等模块里已经间接依赖过这个机制)。但一个**普通 Python list** 对 `nn.Module` 来说只是一个"看起来无害"的属性,`nn.Module` 不会去检查 list 内部装的是不是子模块。这意味着如果直接用普通 list 存这些 block,`model.parameters()`、`model.to(device)`、`model.state_dict()` 这些方法会不会正确地找到 list 里每个 `TransformerBlock` 的参数?

PyTorch 为此专门提供了一个容器类:去查一下 `torch.nn.ModuleList` 的文档,看看它和普通 `list` 的行为差异,以及为什么它能解决上面这个问题。

### `lm_head` 用哪个模块?

`lm_head` 的作用是"把 `d_model` 维的向量变成 `vocab_size` 维的 logits"——这跟你的 `Embedding` 模块(`vocab_size × d_model` 的查表操作)刚好是**方向相反**的两件事。看一下 `run_transformer_lm` 的 weights key 里 `lm_head.weight` 的形状是 `(vocab_size, d_model)`,和 `token_embeddings.weight` 形状一样——但两者的**用法**(forward 里怎么用这个权重矩阵)完全不同,想一下该用你写好的哪个模块类。

## 建议的实现步骤拆分

1. 先写 `TransformerBlock`:构造函数创建两个 `RMSNorm`、一个 `multihead_self_attention`(带 RoPE)、一个 `Feedforward`;`forward` 按上面两条公式顺序算,注意残差连接加的是谁。
2. 单独跑通 `test_transformer_block`(先不管权重命名映射的问题,可以后面再抠 adapter 细节)。
3. 再写 `TransformerLM`:`Embedding` → `nn.ModuleList` 装 `num_layers` 个 `TransformerBlock` → 最终 `RMSNorm` → `lm_head`。
4. 跑通 `test_transformer_lm`。

## 建议的验证方式

- 检查每一步输出的形状:`TransformerBlock.forward` 输入输出形状应该完全一致(`(..., seq_len, d_model)`),因为残差连接要求两者能相加;`TransformerLM.forward` 最终输出应该是 `(batch_size, seq_len, vocab_size)`。
- 单独测试:构造一个 `num_layers=1` 的小模型,确认整个链路能跑通,再测试更多层。
- 想一下:如果 `TransformerBlock` 内部忘了某一步的残差连接(比如漏加了 `x`),从 loss/输出数值上大概会有什么表现?这类"漏加残差"的 bug 往往不会报错,只会让训练变得异常不稳定——这也是为什么建议先靠 shape 检查 + 单元测试(`test_transformer_block`)确认正确性,而不是等到训练阶段才发现问题。
